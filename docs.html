Nexus

The open-source desktop AI Agent platform built with Tauri 2.x + React + TypeScript. The only local-first agent with 40+ built-in tools, multi-agent orchestration, and transparent execution — all running on your machine with zero cloud dependencies.

Get Started → Download Desktop View on GitHub

What is Nexus?

Nexus is a desktop AI Agent platform that runs entirely on your machine. Unlike cloud-based AI tools, Nexus gives you full control over your data, API keys, and agent behavior. It's built with Tauri 2.x for native performance, React for the UI, and a TypeScript sidecar engine that handles all agent logic.

Key differentiators from other AI tools:

  • Local-first — all data stored in SQLite on your machine, zero telemetry
  • 40+ built-in tools — web search, file ops, code execution, browser automation, and more
  • Multi-agent orchestration — delegate tasks to specialized sub-agents
  • Transparent execution — every tool call is visible with inputs, outputs, and timing
  • BYOK (Bring Your Own Key) — use your own API keys, pay providers directly
  • Multi-provider — OpenAI, Anthropic, Google, LM Studio, Ollama, or any custom endpoint

# Installation

Windows

Download the standalone nexus.exe from GitHub Releases. No installer needed — just run the executable. The app is a single file with all dependencies bundled.

Build from Source

For development or customization, build Nexus from source:

bash 
# Clone the repository
git clone https://github.com/DucklingGod/nexus.git
cd nexus

# Install dependencies
npm install

# Run in development mode (opens the window)
npm run tauri dev

# Build standalone executable
npm run tauri build

Prerequisites

  • Node.js 18+ — JavaScript runtime
  • Rust toolchain — for Tauri's native layer (install via rustup)
  • MSVC Build Tools (Windows only) — C++ compiler for native modules
  • Git — for cloning the repository
Rust PATH: On Windows, the Rust toolchain may not be on your PATH automatically. Run export PATH="$HOME/.rustup/toolchains/stable-x86_64-pc-windows-msvc/bin:$PATH" before building.

Verifying the Build

bash 
# Verify frontend compiles
npm run build

# Verify Rust compiles
cargo check --manifest-path src-tauri/Cargo.toml

# Run engine tests (44 tests)
cd engine && npm test

After launching Nexus, you'll see the three-panel layout: Skills panel (left), Chat console (center), and Provider config (right). Try these to get started:

json

      "name": "OpenAI",
      "apiFormat": "openai",
      "baseUrl": "https://api.openai.com/v1",
      "apiKeyRef": "os-keychain",
      "models": ["gpt-4o", "gpt-4o-mini"]
    },
    {
      "name": "LM Studio",
      "apiFormat": "openai",

  "defaultModel": "gpt-4o",
  "theme": "dark",
  "window": {
    "width": 1200,
    "height": 900,
    "decorations": false
  }
}

Provider Auto-Detection

Nexus automatically detects the API format based on the baseUrl:

api.openai.comOpenAI format (GPT-4o, GPT-4, o1, o3)
api.anthropic.comAnthropic format (Claude 4, Claude 3.5)
generativelanguage.googleapis.comGoogle format (Gemini 2.5, Gemini 2.0)
localhostOpenAI-compatible (LM Studio, Ollama)
Any other URLOpenAI-compatible (custom endpoints)
API keys are stored in your OS keychain — never in config files, logs, or exports. Nexus uses os-keychain as the apiKeyRef to reference the key stored securely by your operating system.

# Chat Interface

The main chat console is where you interact with your AI agents. It supports real-time streaming, markdown rendering, and transparent tool execution.

Layout

Nexus uses a three-panel layout:

  • Left panel — Skills manager: browse, enable/disable, and configure skills
  • Center panel — Chat console: your conversation with streaming responses
  • Right panel — Provider config: select providers, models, and API keys

Features

  • Streaming responses — tokens appear in real-time as the LLM generates them
  • Markdown rendering — full support for bold, italic, code blocks, tables, lists, and blockquotes
  • Auto-scroll — follows the conversation when you're near the bottom; pauses if you scroll up to read
  • Tool transparency — every tool call is shown with its name, inputs, outputs, and execution time. Click to expand.
  • Multi-conversation — switch between chat threads with full history persisted in SQLite
  • Message persistence — all messages survive app restarts

Creating Custom Skills

Create a new skill by adding a SKILL.md file to the skills directory:

markdown 
# ~/.nexus/skills/my-skill/SKILL.md
---
name: my-custom-skill
description: A brief description of what this skill does
trigger: When the user asks about X or needs Y done
category: custom
---

## Steps
1. First, analyze the user's request to understand the goal
2. Gather necessary information from available sources
3. Execute the appropriate actions in sequence
4. Verify the results meet the user's expectations
5. Summarize what was done and any follow-up needed

## Pitfalls
- Don't skip input validation — always check edge cases
- If a tool fails, try an alternative approach before giving up
- Keep the user informed of progress on long-running tasks

## Verification
- Confirm the output matches the user's intent
- Check for common errors in the result
- Offer to iterate if the result isn't satisfactory
Auto-skill creation: Nexus can automatically create skills from successful task completions. When it detects a repeatable pattern, it suggests saving it as a skill for future use.

# Conversations

Nexus supports multiple conversation threads, each with full history persisted in SQLite. Your conversations survive app restarts.

Managing Conversations

  • Create — click the "+" button or press Ctrl+N
  • Switch — click a conversation in the sidebar or use Ctrl+K to search
  • Delete — right-click a conversation and select "Delete"
  • Export — export conversations as JSON for backup or analysis

ID System

Each message gets a unique ID based on Date.now() to prevent collisions with database-loaded messages after app restarts. This ensures new messages never overwrite existing ones.

# Smart Router

The smart router is Nexus's brain for provider selection. It automatically picks the best model for each task based on complexity, context window needs, and cost optimization.

How It Works

  1. Analyze — parses the user's request for complexity, domain, and context requirements
  2. Match — checks available providers and their capabilities (context window, cost, speed)
  3. Select — picks the optimal model: complex tasks get GPT-4o/Claude, simple tasks get GPT-4o-mini
  4. Fallback — if a provider returns a rate limit (429) or error (503), automatically retries with the next best option
  5. Learn — tracks success rates and latency to improve future routing decisions

Routing Rules

Task TypeRoute ToReason
Simple Q&A, formattingGPT-4o-mini / Gemini FlashFast, cheap, sufficient quality
Complex reasoning, codingGPT-4o / Claude SonnetBest reasoning, larger context
Creative writing, analysisClaude Opus / GPT-4oSuperior writing quality
Privacy-sensitive tasksLM Studio / OllamaFully local, zero data leaves machine

# Semantic Cache

Nexus caches semantically similar queries to reduce API costs and improve response times. Unlike exact-match caches, the semantic cache uses vector embeddings to find similar past queries.

Configuration

json 
{
  "cache": {
    "enabled": true,
    "similarityThreshold": 0.92,
    "maxEntries": 1000,
    "ttlHours": 24,
    "embeddingModel": "text-embedding-3-small"
  }
}
ParameterTypeDefaultDescription
enabledbooleantrueEnable or disable the semantic cache
similarityThresholdnumber0.92Minimum cosine similarity to consider a cache hit (0-1)
maxEntriesnumber1000Maximum number of cached entries before LRU eviction
ttlHoursnumber24Time-to-live for cache entries in hours
embeddingModelstringtext-embedding-3-smallOpenAI embedding model for vectorization
Cost savings: The semantic cache can reduce API costs by 30-50% for repetitive workflows. Similar queries like "search for AI news" and "find latest AI articles" will hit the same cache entry.

# Prompt Caching

Nexus leverages provider-side prompt caching to reduce costs and latency. When available (OpenAI, Anthropic), repeated system prompts and context are cached by the provider, so you only pay for new tokens.

How It Works

  • Automatic — Nexus automatically prefixes conversations with cache-friendly markers
  • Provider support — works with OpenAI's prompt caching and Anthropic's cache breakpoints
  • Cost reduction — cached tokens are billed at 50% (OpenAI) or 90% discount (Anthropic)
  • Zero config — no setup needed, Nexus handles it transparently

# Transparency

Every action Nexus takes is visible and auditable. Unlike black-box AI tools, Nexus shows you exactly what's happening under the hood.

What's Visible

  • Tool calls — name, input parameters, output, and execution time for every tool invocation
  • Provider selection — which model was chosen and why (smart router decisions)
  • Cache hits — when a response was served from semantic or prompt cache
  • Token usage — input/output tokens and estimated cost per request
  • Error handling — retries, fallbacks, and failure reasons

Chips & Badges

Tool calls are displayed as expandable "chips" in the chat. Each chip shows:

  • 🔧 Tool name — e.g., "web_search", "terminal", "read_file"
  • ⏱️ Duration — how long the tool took to execute
  • 📊 Status — success (green) or error (red)
  • 📋 Details — click to expand and see full input/output

# Tauri Bridge

The Tauri bridge connects the React frontend to the Rust backend and the TypeScript sidecar engine. It handles:

  • IPC commands — invoke() from frontend, #[tauri::command] in Rust
  • Event system — listen() for streaming responses and real-time updates
  • Window management — frameless window (1200×900), custom title bar, dark theme
  • Keychain access — secure API key storage via the OS keychain
  • SQLite bridge — database operations for conversations and settings
  • Sidecar lifecycle — start, stop, and communicate with the TypeScript engine

Communication Flow

typescript 
// Frontend → Tauri → Sidecar
const response = await invoke('chat_send', {
  message: "Search for AI news",
  conversationId: 42
});

// Sidecar → Frontend (streaming)
listen('chat:delta', (event) => {
  // event.payload = { text: "Here are..." }
  appendToken(event.payload.text);
});

// Sidecar → Frontend (tool calls)
listen('chat:tool', (event) => {
  // event.payload = { name: "web_search", input: {...}, output: {...} }
  showToolCall(event.payload);
});

# TypeScript Sidecar

The TypeScript sidecar is the brain of Nexus. It runs as a child process and handles all agent logic, tool execution, and provider communication.

Components

SmartRouterSelects the best provider and model for each task
ToolRegistryManages 40+ tools with category, validation, and execution
SkillsEngineLoads, matches, and applies skills based on user intent
SemanticCacheVector-based query caching for cost reduction
PromptCacheProvider-side prompt caching for repeated contexts
TransparencyLayerLogs all tool calls, decisions, and timing for auditability
StreamHandlerReal-time SSE streaming from providers to frontend

JSON-RPC Protocol

Communication between Tauri and the sidecar uses JSON-RPC 2.0 over stdio:

json 
// Request (Tauri → Sidecar)
{
  "jsonrpc": "2.0",
  "method": "chat.send",
  "params": {
    "message": "Search for AI news",
    "conversationId": 42
  },
  "id": 1
}

// Response (Sidecar → Tauri)
{
  "jsonrpc": "2.0",
  "result": {
    "stream": true,
    "streamId": "s_abc123"
  },
  "id": 1
}

Nexus uses SQLite for all local storage via better-sqlite3. Vector operations use sqlite-vec for the semantic cache.

Never store API keys in the database. Keys are stored in the OS keychain (Windows Credential Manager, macOS Keychain, Linux Secret Service). The apiKeyRef field in config references the keychain entry, not the key itself.

# Security Model

Nexus follows a zero-trust, local-first security model:

Principles

  • No telemetry — zero data is sent to Nexus servers. No analytics, no phone-home, no tracking.
  • OS keychain — API keys stored in the operating system's secure keychain, never in files.
  • Local storage — all data (conversations, settings, cache) stored in SQLite on your machine.
  • No cloud dependencies — the app works fully offline with local models (LM Studio, Ollama).
  • Transparent execution — every tool call is logged and visible to the user.
  • Sandboxed tools — code execution runs in isolated environments with configurable permissions.

API Key Handling

  1. User enters API key in the Provider Config panel
  2. Key is immediately stored in the OS keychain
  3. Config file stores "apiKeyRef": "os-keychain" — a reference, not the key
  4. At runtime, the sidecar reads the key from the keychain for each API call
  5. Keys are never logged, exported, or included in conversation history

# Troubleshooting

Common Issues

Blank window on startup

The frontend takes a few seconds to load. If the window stays blank for more than 10 seconds, check that the dev server is running (npm run dev) or that the build output exists in dist/.

"API key not found" error

Make sure you've added your API key via the Provider Config panel. Keys are stored in the OS keychain — if you're on Linux, ensure secret-service is available.

Tools not executing

Check that the TypeScript sidecar is running. Look for the sidecar process in your task manager. If it's not running, restart Nexus.

Slow responses

If responses are slow, try:

  • Switching to a faster model (e.g., GPT-4o-mini instead of GPT-4o)
  • Using a local model via LM Studio or Ollama
  • Enabling the semantic cache for repeated queries
  • Checking your internet connection for cloud providers

Sidecar path issues after moving project

The sidecar path is baked at compile time. If you move the project directory, you need to rebuild: npm run tauri build

Need help? Open an issue on GitHub Issues with your error message, OS version, and Node.js version.
